# AI 팁 섹션 — 대상 독자와 스코프 (결정 기록)

## 배경

`TIP_CATEGORIES`(`summarizer/claude_summarizer.py`)는 2026-03-27 "생활밀착" 설계 →
`fd8014d` 마케팅·이커머스 개편 → `c813478` all-Claude-Code 개편까지 두 번 무문서로
피벗했다. 그 결과 뉴스 본문(`summarize()` 프롬프트, `:96-99`)은 독자를 "일반 독자
(비개발자 직장인·소비자)"로 정의하면서, 팁 섹션은 GitHub PR 리뷰·GitHub Actions
같은 개발자 전용 작업을 권하는 자기모순이 생겼다 (Fable 5 리뷰로 확인, 2026-07-08).
이 문서는 그 리뷰를 반영해 확정한 방향을 기록해, 다음 피벗이 다시 무문서로
일어나지 않게 한다.

## 대상 독자

30-40대 비개발자 직장인·소비자. 코딩 경험 없음을 기본 전제로 한다.
(`summarize()`의 채점 기준과 동일 인물상 — 두 섹션이 같은 독자를 가정해야 한다.)

## 팁이 파는 것

"오늘 당장 써먹을 수 있는 AI 활용법 1개." 학습 커리큘럼이 아니라 매회 독립된
실용 팁이다. 단, 완전히 무작위는 아니고 최근 반복은 피한다 (`TIP_HISTORY_SIZE`,
`main.py`).

## difficulty 이원화 원칙

- `prompt-only`: Claude.ai 웹에 붙여넣기만 하면 끝. 설치·앱 실행 없음.
- `claude-code`: Claude Code 데스크탑 앱 필요 (파일시스템 접근, 스크립트 생성·실행,
  반복 자동화처럼 웹 채팅만으로는 안 되는 작업에만 사용).

카테고리를 추가할 때 실제로 로컬 파일시스템/스크립트 실행이 필요한지 먼저
판단할 것. "자동화"라는 이름이 붙어도 웹 채팅에 텍스트를 붙여넣어 되는 작업이면
`prompt-only`로 분류한다 (2026-07-08 리뷰에서 회의록 정리·정기 보고서·문서 추출
3개를 이 기준으로 재분류함).

뉴스레터 렌더링(`newsletter.py` tip_html)은 `difficulty`를 독자용 배지
("💻 Claude Code 앱 필요" / "🌐 웹에서 바로")로 노출한다 — 내부 분기값이자
독자에게 보이는 기대치이므로 둘을 어긋나게 두지 말 것.

## 개발자 전용 카테고리를 제거한 이유

GitHub PR 리뷰, GitHub Actions 자동화, 커밋/CHANGELOG, MCP 연동, CLAUDE.md 설정,
Anthropic SDK("사내 Claude AI 붙이기") — 이 6개는 대상 독자에게 애초에 존재하지
않는 문제(PR, 리포지토리, CI 파이프라인 등)를 전제로 해 제거했다. 개발자 대상
콘텐츠가 필요하면 이 뉴스레터가 아니라 별도 트랙으로 분리할 것 — 같은 발행물
안에서 독자 정의를 두 개 갖지 않는다.

## 온보딩

`last_tip_category.txt`가 없는 첫 실행(신규 구독자 관점의 "첫 팁")은
`ONBOARDING_TIP_CATEGORY`("긴 문서 핵심 요약", `claude_summarizer.py`)로 고정한다 —
설치도, 파일 업로드도 필요 없는 가장 진입장벽 낮은 팁으로 시작해야 두 번째 팁부터
이탈하지 않는다.

## 앞으로 카테고리를 바꿀 때

이 문서의 "대상 독자"·"difficulty 이원화 원칙"과 충돌하는 카테고리를 추가하기 전에
이 문서를 먼저 갱신할 것. 문서 없는 피벗이 지금 문제의 원인이었다.
